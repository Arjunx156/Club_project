ORCA Specification

ORCA (Odyssey Research & Club of AI) is a monolithic portal supporting member authentication, event registration, and AI integration. We abandoned microservices. State lives in PostgreSQL via Supabase. Compute is serverless Flask. The system is designed to be fully synchronous and predictable, handling constrained payloads without complex distribution layers.

§1. Data Flow & Monolith

The architecture is exactly one tier. The frontend is vanilla HTML and JavaScript served directly by a Flask monolith on Vercel. We do not use React. We do not use Webpack. The DOM is manipulated directly. This yields zero build times and zero client-side hydration overhead.

State synchronization happens exclusively at the API boundary. When a user requests to join a waitlist, the client issues an XHR POST, and the server executes a parameterized SQL transaction against PostgreSQL. The schema is normalized; we do not denormalize event counts. Instead, we query aggregates on read. This trades a slight read latency penalty for absolute write consistency.

§2. State Resolution & Constraints

Event registration is a hard constraint problem. Each event has a scalar capacity C. A user is either registered, waitlisted, or unregistered. The state transition must be atomic to prevent overbooking and race conditions under concurrent load.

We implemented a deterministic status resolution. When capacity is exceeded, the state flag transitions to a waitlist status based purely on the insertion timestamp. There is no background worker. Promotion off the waitlist happens eagerly at the moment of cancellation via a transaction block in the Flask handler.

P(promotion) = if rank <= C - active_registrations: 1.0 else: 0.0 where rank = position(timestamp) in {t_1, t_2, ... t_N}

§3. Intelligence Integration

The AI assistant is stateless. Every prompt submitted by the user is packaged with exactly two pieces of context: the user's club role and the current route's semantic footprint. We use Google Gemini 2.5 Flash as the reasoning engine. We do not maintain conversational memory on the server. The client holds the transcript and replays the window on each request.

This approach limits context length to the model's token limit but eliminates database roundtrips for chat history. The prompt engineering is severe. We instruct the model to return raw JSON or bare text. No markdown formatting unless explicitly requested. This keeps parsing robust and prevents client-side rendering vulnerabilities.

def generate_response(prompt, context):
    payload = {
        "model": "gemini-2.5-flash",
        "contents": [prompt, context],
        "generationConfig": { "temperature": 0.1 }
    }
    return execute_inference(payload)

§4. Computational Complexity

We optimize for strict upper bounds on execution time. The serverless environment penalizes long-running tasks. Below is the complexity matrix for critical path operations.

Algorithm Time Complexity Reference
Waitlist Insertion O(1) app.py:register_event
Status Resolution O(N log N) db/schema.sql
Semantic Context Fetch O(1) static/js/app.js
AI Payload Embedding O(L²) Gemini API

§5. Infrastructure & Phasing

Deployment is deterministic via Vercel. The WSGI entrypoint configures routing dynamically based on vercel.json. The environment footprint is minimal: connection strings and API keys. Connection pooling is handled by Supabase at the edge (TCP port 6543) because Vercel serverless functions do not persist database connections across invocations.

Phase 1: Persistence

Supabase PostgreSQL integration. Transaction pooler configuration. JWT session boundaries and strict RBAC enforcement.

Phase 2: Constraints

Waitlist queuing logic. Capacity constraints resolution. UI status reflection and state synchronization.

Phase 3: Intelligence

Gemini integration via Flash API. Semantic context injection. Stateless token processing and JSON extraction.
